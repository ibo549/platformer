## IMPORTANT: File Editing Rules

- **Always edit `game.src.html`** — this is the canonical source (~98KB, readable).
- **Never edit `game.html` directly** — it is a build artifact generated by `./build.sh`.
- **Never edit `PlatformerApp/PlatformerApp/Resources/game.html`** directly. It is auto-copied from root by `deploy.sh`.
- After editing `game.src.html`, run `./build.sh` to regenerate `game.html`.
- Run `./deploy.sh` to build and deploy (it runs `build.sh` automatically).

## Deploying to iPad/iPhone

### Quick Deploy (one command)

```bash
cd ~/Desktop/platformer
./deploy.sh
```

This script automatically:
1. Runs `build.sh` to regenerate `game.html` from `game.src.html` + sprites
2. Syncs `game.html` from project root into `Resources/`
3. Detects the USB-connected device
4. Clean builds the Xcode project
5. Adds icons and patches `CFBundleIcons` in Info.plist
6. Re-signs the app bundle
7. Installs on the device

### Prerequisites
- Xcode 26 installed (for SDK and toolchain)
- Device connected via USB cable
- Developer Mode enabled on device: Settings > Privacy & Security > Developer Mode
- Device trusted: tap "Trust" when prompted on device
- ideviceinstaller installed: `brew install ideviceinstaller`

### First Launch on New Device
Settings > General > VPN & Device Management > tap your developer profile > Trust

## Known Issues & Workarounds

### Xcode 26 codesign "code object is not signed at all" for embedded.mobileprovision
**Fix:** Add `OTHER_CODE_SIGN_FLAGS = "--deep"` in Xcode build settings, or pass `--deep` when manually signing.

### ibtool hangs on LaunchScreen.storyboard
**Fix:** Remove storyboard from build phase. Do NOT include .storyboard files in the Resources build phase.

### actool hangs on Assets.xcassets
**Fix:** Remove asset catalog from build phase. Add app icons manually to the .app bundle and set CFBundleIcons in Info.plist via PlistBuddy.

### iOS app bundle must be flat (no Resources subdirectory)
**Fix:** Use individual file references in the Xcode project, not a folder reference. Files must be at the root of the .app bundle, not in a `Resources/` subdirectory. Otherwise iOS installd fails with "Missing bundle ID".

### WKWebView canvas taint with file:// URLs
**Problem:** WKWebView treats file:// images drawn to canvas as cross-origin, causing WebGL `texImage2D` to fail silently (black box). The `createImageBitmap` with crop parameters and `imageOrientation` options is also unreliable across iOS versions.
**Fix:** Embed sprite sheets as base64 data URLs directly in game.html. Data URLs are same-origin, so canvas-based texture cropping works perfectly in WKWebView. The game.html grows to ~10MB but this is fine for an embedded app. The PNG files in Resources/ are no longer needed at runtime but kept for asset regeneration.

### DDI not found by devicectl
**Fix:** Use `pymobiledevice3 mounter auto-mount` instead of `devicectl manage ddis update`.

## Game Features
- Three playable characters: Halil (blue theme, Mickey companion), Lara (red/pink theme, Minnie companion), and Karolcia (dog companion)
- 6-frame run animation: 2 sprite frames + 4 derived frames (squash/stretch/lean) generated at load time via `deriveRunFrame()`
- Shield power-up with pixel heart animation (blue for Halil, red for Lara); HUD shows shield emoji icon at end of row with colored border glow + pulse animation (doesn't shift other HUD elements)
- Shield coins (blue for Halil, red for Lara)
- Touch controls with visual indicators for iPad/mobile
- Confetti win animation, lives system, enemy obstacles
- Game over and win screens return to character/difficulty selection (not direct restart)
- Adaptive pixel art rendering using physical pixels (CSS size * devicePixelRatio) for clean integer upscaling:
  - iPad: PIXEL_SCALE 3 (~512px internal height, 6x physical upscale)
  - iPhone portrait: PIXEL_SCALE 2 (~466px internal height, 6x physical upscale)
  - iPhone landscape: PIXEL_SCALE 1 (native CSS resolution, 3x physical upscale)
  - Recalculates on orientation change
- iPad: landscape-only. iPhone: portrait + landscape (rotates freely)
- Three difficulty modes (selected on start screen after character select):
  - **Easy:** Only brown poop enemies
  - **Normal:** Brown + Stick + Strawberry enemies (no pterodactyl)
  - **Hard:** All enemies including pterodactyl (full difficulty)
- Responsive start screen: `viewport-fit=cover` for full-screen coverage on notch/Dynamic Island devices; compact layout via `@media (max-height: 500px) and (orientation: landscape)` for iPhone landscape

## Enemies
- **Brown enemy** — ground-based, 1 stomp to kill, random jumping
- **Stick enemy** — ground-based, 2 stomps to kill (first stomp cracks it)
- **Strawberry enemy** — ground-based, 1 stomp to kill, "GAW GAW" speech
- **Pterodactyl** — flying enemy, purple/indigo color scheme, 2 stomps to kill:
  - Patrols left/right with sine-wave vertical bob and wing flap animation
  - Shoots bright flame projectiles (white-hot core + yellow glow halo) with screech sound (`sfxPteroScreech`)
  - First stomp: deforms (squashed body, droopy wing, slower, flies lower) + yellow stars orbit head + dazed sound (`sfxPteroDazed`)
  - Second stomp: killed with particle burst, +40 score
  - "SKREEEE!" speech bubble appears when near player or firing

## Water Puddles
- Translucent blue puddles with brown mud banks, white specular highlights, and animated ripple dots
- ~8 puddles scattered across the level on the ground
- **Player:** splash sound (`sfxSplash`) when running through (bandpass-filtered noise + descending droplet tones)
- **Ground enemies:** two-encounter melt system — only triggers when both puddle and enemy are on-screen:
  - **First puddle:** enemy gets visibly wet (blue tint, 5 orbiting water droplets, splash sound) but keeps walking
  - **Second puddle:** enemy stops moving, shrinks over 5 seconds, turns blue, says "oh my god-tow"
  - Re-entering the same puddle counts as a second encounter
  - Water melt deaths do NOT count towards kill counter (no extra lives); stomping a melting enemy yourself does count
- Enemy patrol ranges increased (brown: ±6-10, stick: ±6-10, strawberry: ±7-11 units) so they encounter puddles more often

## Progression
- 5 enemy kills = 1 extra life (shown as 16px poop emoji pips in HUD)
- Eggs never spawn on spike hazards (nudged 3 units away)

## Adding a New Character

### Sprite sheet requirements

- **Format:** PNG, 1536x1024 pixels (same as existing sheets)
- **Layout:** 6 frames arranged left-to-right in a single row: idle, run1, run2, jump, fall, land/crouch
- **Naming:** `<name>.png` in the project root (lowercase, e.g. `mario.png`)
- **Orientation:** Character should face RIGHT. If your art faces left, set `mirror: true` in `characterData` and the engine will flip it
- **Background:** Transparent (alpha channel). The character is rendered as a textured plane with `alphaTest: 0.1`
- 4 extra run frames (contact/push-off variants) are auto-generated at load time from run1 and run2 via `deriveRunFrame()` — no need to draw them

### Step-by-step process

All edits are in `game.src.html` unless noted.

#### 1. Determine crop coordinates

Open the PNG in an image editor and find:
- **contentTop / contentBottom:** The Y pixel range containing the character across all frames (trim empty space above/below)
- **crops:** For each of the 6 frames, find `{ x, w }` — the X offset and width of that frame

The preview canvas height should be `contentBottom - contentTop`.

#### 2. Add to `characterData` (~line 692)

```js
newchar: {
  mirror: false,  // set true if sprite faces left
  file: '{{NEWCHAR_SPRITE}}',
  contentTop: <top>,
  contentBottom: <bottom>,
  crops: [
    { x: <idle_x>,  w: <idle_w> },   // idle
    { x: <run1_x>,  w: <run1_w> },   // run 1
    { x: <run2_x>,  w: <run2_w> },   // run 2
    { x: <jump_x>,  w: <jump_w> },   // jump
    { x: <fall_x>,  w: <fall_w> },   // fall
    { x: <land_x>,  w: <land_w> },   // land/crouch
  ],
  textures: [],
  loaded: false,
},
```

#### 3. Add character select UI (~line 276)

Add a new `<div>` inside `<div class="char-select">`:

```html
<div class="char-option" data-char="newchar" id="char-newchar">
  <canvas class="char-preview" id="preview-newchar" width="200" height="<contentBottom - contentTop>"></canvas>
  <span class="char-name">NEWCHAR</span>
</div>
```

#### 4. Load and preview the character (~line 815)

Add two lines:

```js
loadCharacter('newchar');
drawPreview('newchar');
```

#### 5. Create the base64 sprite file

```bash
cd ~/Desktop/platformer
python3 -c "
import base64
with open('newchar.png', 'rb') as f:
    b64 = base64.b64encode(f.read()).decode()
with open('sprites/newchar.b64', 'w') as f:
    f.write('data:image/png;base64,' + b64)
print(f'Written sprites/newchar.b64 ({len(b64)} chars)')
"
```

#### 6. Add placeholder to `build.sh`

Add a new entry to the `placeholders` dict:

```python
"{{NEWCHAR_SPRITE}}": root / "sprites" / "newchar.b64",
```

#### 7. Set character-specific theming

Search for `selectedChar ===` to find all character-specific branches. These need updating:

| Location | What it controls | How to find |
|----------|-----------------|-------------|
| `buildShieldHearts()` (~line 965) | Shield heart color (blue for halil, red for others) | `selectedChar === 'halil' ? 0x44aaff : 0xff2255` |
| `activateShield()` (~line 998) | Shield HUD background color | `selectedChar === 'halil'` |
| `createCompanionMesh()` (~line 2520) | Which companion follows the player (Mickey/Minnie/Scooby) | `if (selectedChar === 'halil')` |
| `createRedCoin()` (~line 2776) | Shield coin color | `selectedChar === 'halil'` |
| `selectCharacter()` (~line 881) | Companion HUD icon emoji | `iconMap = { halil: '🐭', ... }` |
| Win text (~line 3283) | "ALL 8 ___S RESCUED!" | `selectedChar === 'halil' ?` ternary chain |

#### 8. Create a companion mesh (optional but expected)

Each character has a companion that hatches from eggs. Write a `createNewcharCompanionMesh()` function following the pattern of `createMickeyMesh()` / `createMinnieMesh()` / `createScoobyMesh()`. Then add a branch in `createCompanionMesh()`.

#### 9. Build and test

```bash
./build.sh        # Regenerate game.html
./serve.sh        # Test in browser
./deploy.sh       # Deploy to device
```

### Watchouts

- **Canvas taint on iOS:** Sprite sheets MUST be embedded as base64 data URLs in game.html. External file:// images drawn to `<canvas>` are treated as cross-origin by WKWebView, causing black/broken textures. Never try to load sprites via `<img src="file://...">` and draw to canvas.
- **Crop accuracy matters:** If crops are off by even a few pixels, the character will jitter between frames as it shifts within the normalized NORM_W x NORM_H canvas. Check alignment by cycling through all 6 frames in idle.
- **Preview canvas height:** The `height` attribute on the preview `<canvas>` element must match `contentBottom - contentTop` exactly, or the character select preview will be stretched/squashed.
- **Mirror flag:** If set, the entire sprite sheet is flipped horizontally during loading. All crop X coordinates are still relative to the original (un-flipped) image.
- **File size:** Each character's base64 adds ~3-3.5MB to game.html. This is fine for the embedded WKWebView app but be aware the file grows.
- **Companion required:** The game expects 8 eggs to hatch per level. Without a companion mesh for your character, eggs will spawn but nothing will appear when hatched.
- **generate_icon.py / generate_splash.py:** Add the new character's crop data to the `CHARACTERS` dict in both scripts so you can generate icons/splash with the new character.

## Build System
- `game.src.html` (~100KB) is the editable source with `{{HALIL_SPRITE}}`, `{{LARA_SPRITE}}`, `{{KAROLCIA_SPRITE}}` placeholders
- `sprites/*.b64` files contain the raw base64 data URL strings for each character's sprite sheet
- `build.sh` (Python3) replaces placeholders with `.b64` file contents → outputs `game.html` (~10MB)
- `game.html` is committed for easy browser testing but is always regenerated from source
- `deploy.sh` calls `build.sh` automatically before building the Xcode project
